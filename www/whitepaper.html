Provable Simulation as Verifiable Decision Infrastructure

LL BERK Yazılım Teknolojileri A.Ş. · 2026 · v1  |  A technical position paper on deterministic, auditable, forkable simulation.
Abstract. Simulation is increasingly used not to visualize, but to decide — which course of action to take, whether a running system is behaving correctly, whether an autonomous decision was justified. Yet the simulations these decisions rest on are usually built on floating-point arithmetic, which is neither reproducible across machines nor separable from its own numerical drift. This paper argues that a decision-grade simulation needs four properties that ordinary engines lack: it must be deterministic (bit-identical on every platform), replayable, forkable (able to explore alternative futures), and provable (every run cryptographically addressable and tamper-evident). We describe how Berk DT delivers all four from a single fixed-point core, what this unlocks for defense, industry, and robotics — and, just as importantly, what determinism does not guarantee.

1The problem: decisions need simulation you can trust

A physics engine that renders a plausible-looking scene is a solved problem. A simulation whose output you are willing to act on is not. The moment a simulation informs a real decision — engage or hold, ship or halt, trust the sensor or not — three questions become unavoidable:

  • Reproducibility: if we run the same scenario again, on another machine, do we get the same answer?
  • Attribution: when the simulation and reality diverge, is it model error, a real fault, or tampering?
  • Defensibility: can we prove, after the fact, what the system computed and why?

For most simulation stacks, the honest answer to all three is "not really." That gap is not a UX problem; it is the difference between a demo and an instrument.

2Why floating-point simulation fails the trust test

IEEE-754 floating-point results depend on compiler flags, fused-multiply-add availability, instruction ordering, and thread scheduling. The same code can produce slightly different last bits on a different CPU, and those tiny differences compound over a simulation's timeline. Three consequences follow:

1. No cross-platform ground truth. "It worked in our lab" does not mean it works on the deployed hardware, the partner's machine, or next year's compiler. There is no single answer to compare against.

2. Drift masks faults. If the prediction wanders on its own, you cannot set a tight tolerance between predicted and measured state — because you can't tell model drift from a genuine anomaly. You loosen the threshold, and the monitor goes blind exactly when it matters.

3. Nothing to sign. If the result isn't bit-stable, there is no canonical value to hash, log, or certify. "The simulation ran" is a claim, not evidence.

Determinism is often treated as a nice-to-have for lockstep multiplayer. For decision-grade simulation it is the foundation everything else stands on.

3Why replay is necessary — but not sufficient

Record-and-replay is the first thing a serious simulation stack adds: capture the seed and ordered inputs, and you can reproduce a run. This answers "what happened?" and is essential for debugging and audit. But it has two limits. First, on a float engine, replay is only reproducible on the same binary — it is not a cross-platform proof. Second, and more fundamentally, replay only tells you about the timeline that actually occurred. It cannot answer the question decision-makers actually ask:

Not "what happened?" but "what else could have happened — and which choice was best?"

Answering that requires branching the present into alternative futures. Replay is a line; decisions live in a tree.

4Why fork is the missing layer

If a simulation is deterministic and can snapshot/restore its full state, then from any live moment you can fork it: take a snapshot, run branch A under one plan, restore, run branch B under another, and so on. Each branch ends in a state you can score (how good is this outcome?) and address by hash (which exact future is this?). Because the core is deterministic, the same base plus the same plan always yields the same branch — so the set of options you produce is reproducible and defensible, not a one-off Monte-Carlo artifact.

Both are the operational face of the same deterministic core.

5Why hash is the binding that makes it provable

A deterministic run produces, at every step, a 64-bit state_hash — a fingerprint of the full carry-over state. This single value does an enormous amount of work:

  • Cross-platform proof: two machines that agree on the hash agree on every bit of the simulation. CI checks this on Linux, Windows, and macOS on every change.
  • Tamper-evidence: a run report can be sealed with a metadata-seeded hash chain; altering any frame, parameter, or the seal itself breaks verification. The report is proven not by a signature but by re-running the scenario — physics replaces the notary.
  • Addressable futures: each forked branch is identified by its result hash, so a course of action can be cited, compared, and reproduced exactly.
  • Divergence forensics: comparing two per-frame hash traces pinpoints the first frame where two runs (or a run and reality) parted ways.

6How Berk DT delivers all four

A single Rust core, consumed through a C-ABI (plus Unity, Unreal, and WebAssembly bindings), provides:

  • Deterministic: all simulation math is fixed-point (Q32.32) integer arithmetic; trigonometry via CORDIC, square root via a deterministic integer method. No IEEE float touches the state that feeds state_hash.
  • Replayable: seed + ordered input + periodic hash reproduce a run exactly and detect divergence; full snapshot/restore serializes the carry-over state to bytes and back, bit-identically.
  • Forkable: the Advisor (deterministic what-if tree) and rollback netcode both build on the same atomic snapshot path — a branch can never corrupt its base.
  • Provable: cross-platform golden-hash CI, a tamper-evident audit report tool, a divergence debugger, and formal (TLA+) models of the concurrency and decision logic.
The physics breadth (2D/3D rigid bodies, joints, CCD, convex/compound shapes, cloth, fluid, vehicles, character controller, deterministic LiDAR) is real and tested — but it is the substrate, not the product. The product is the four properties above applied to decisions.

7What this unlocks

Defense — decision safety

Course-of-action analysis where the option space is provable rather than a one-off scenario; after-action analysis where a field event can be reproduced bit-for-bit (given deterministically logged inputs) and alternative decision branches explored; air-gapped deployment with no external dependency; auditable autonomy where every decision carries an epistemic state, authority level, and provenance hash.

Industry — operational optimization

An operational future tree for what-if planning (shutdown optimization, supply-chain forks); a shadow twin for predictive maintenance and anomaly/tamper detection with a threshold tight enough to be useful; reproducible audit of every run.

Robotics / AI — sim-to-real verification

Reproducible training and evaluation; deterministic sensor (LiDAR/range) simulation for closed sensor-loop replay; a complementary "ground-truth execution layer" beside GPU training stacks. Train anywhere; execute provably.

8What determinism does not guarantee

The credibility of this entire position depends on being precise about its limits. Overclaiming here is how you lose a technical audience — so we are explicit:

Determinism removes simulation noise, not model error. The Shadow Twin's residual is only as meaningful as the fidelity of the model; the persistent gap between a real asset and its twin (the reality-gap) is managed by calibration, and the alarm threshold is set above that band. Determinism makes the simulation side of the comparison exact — it does not make the model correct.

Performance figures are advisory. They are single-machine measurements; no certified worst-case execution time (WCET) is claimed in public material.

No formal certification yet. DO-178C / ISO 26262 and similar are not held today. The core is designed to produce the evidence artifacts those processes require — hash logs, replay certificate, divergence report, audit trail, requirement traceability, TLA+ models — and to keep the certification path open (stable-only safety core, documented toolchain alignment). "Aligned with" a standard is a claim we do not make without the evidence.

Fork is not free search. The Advisor forks and scores branches deterministically; producing good options still depends on the scoring function and the branch-generation strategy. It is a reproducible evaluation substrate, not an oracle.

These are not weaknesses to hide; they are the boundary that makes everything inside it trustworthy. The strongest version of this pitch is the honest one.

Conclusion

Berk DT is not, at its core, a physics engine, a digital twin, or netcode — those are surfaces. It is a deterministic, auditable, forkable reality engine: an infrastructure for making decisions you can prove. Told as a physics engine, it competes on features it does not need to. Told as verifiable decision infrastructure, it opens a category where the alternative is trust without proof — and in defense, industry, and safety-critical robotics, trust without proof is not enough.
